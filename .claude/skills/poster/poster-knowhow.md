**他のポスターにもそのまま応用できる**知見集。コードで（pptxgenjsで）ポスターを作る際の教科書。

## 1. ワークフロー：作って終わりにしない

```
生成スクリプト実行 → PNG書き出し → 自分の目で見る → 修正 → 再生成 → …収束するまで
```

- **コードが正しくてもデザインは壊れている**。テキストの折返し・要素の重なり・z-order逆転は、レンダリングして見るまで絶対に分からない。
- PNG書き出しは PowerShell + PowerPoint COM（`export-slides.ps1`）で自動化済み。
- **ユーザーに見せるのは「インパクト・創意性まで仕上がってから」**。技術的に破綻していないだけの状態で見せない。

### 目視QAチェックリスト（毎サイクル確認）
1. テキストの**意図しない折返し**（「！」「。」だけ2行目に落ちる等）→ `wrap: false` で対策
2. 要素同士の**重なり・衝突**（特に価格と擬音、バッジと画像）
3. **z-order**：後から `add` したものが前面。バッジを画像より先に描くと隠れて消える
4. **フレーム系素材との干渉**：枠画像の縁は思ったより太い。縁の実測をしてから内側に安全域を設定
5. **断ち落とし**のつもりが「見切れ」になっていないか（重要文字が端で切れるのはNG）
6. 縮小表示（サムネイル）で見て、**一番伝えたい情報が最初に目に入るか**

## 2. レイアウト原則

- **視線誘導はZパターン**：左上（キャッチ）→右上→中央（メインビジュアル）→左下→右下（行動情報）。
- **「上に写真／中にキャッチ／下に情報」の凡庸な三段構成に陥らない**。斜め配置・断ち落とし・特大タイポ・非対称で構図に動きを出す。
- **重心を決める**：主役（手元・商品・顔）を「画面全体の中心」に数式で合わせる。
  画像内の注目点の位置比率（focusRatio）から逆算する：
  `imgX = slideCenterX - boxW * focusXRatio`
- **情報量は極限まで削る**。日付・場所・価格・無料トッピング、以上。書きたいことはコピー1本に集約。
- 要素を**少し回転（±3〜8°）**させるだけで勢いが出る。全部水平は死ぬ。
- 情報の再構成はユーモアの源泉：選挙ポスター風なら屋台情報を「公約」「投票日」「投票所」に言い換える等、**テーマの文法に情報を翻訳する**と一気に独創的になる。

## 3. 配色

- **3色まで**（メイン・サブ・アクセント）＋比率 **70:25:5**。
- 食欲を刺激するのは**赤・黄・オレンジの暖色**。青は清涼・冷静（食品ならかき氷等の例外のみ）。
- 「安っぽい」と感じたら原因は大抵 **フラットな単色塗り＋多色混在**。色数を絞って質感（写真・実測ベースの枠）を足す。
- 実績パレット：
  - ダイナー: 赤`D42B2B`×黒`1A1A1A`×ターコイズ`40E0D0`
  - ネオン: 黒地`0A0012`×シアン`00F5FF`×マゼンタ`FF2D78`×ライム`CCFF00`
  - POP特価: ビビッド赤`D32F2F`×黄`FFEB3B`×黒`212121`（B案・インパクト最強）
  - チーズ: カスタード`FFE5A3`×サンビーム`FCDB66`×アンバー`F19A3A`×ティンバー`A8785D`
  - 選挙: 白×紺`1B3F8F`×赤`D7000F`

## 4. タイポグラフィ

- **太字にはクッキリした「フチ」**（オフセット複製8方向）。**ぼかし影とフチを重ねると安っぽくなる**——どちらか一方。
- 価格・訴求ポイントは**恥ずかしいくらい大きく**（本文の3倍以上）。強弱をつける（¥300を¥200より大きく等、売りたい方を強調）。
- オノマトペ（じゅわっ・パリッ・とろ〜り）は**独立したビジュアル要素**として斜めに大きく置く。引き出し線でつなぐと台無し。
- 選挙・和風など日本語主体のテーマは `BIZ UDGothic` 等の和文フォント。ポップ系はImpact（英数字のみ効く。日本語はフォールバックするのでサイズ感を実測で確認）。
- 名前・見出しの**ひらがな化**は親しみやすさの definitive な手法（選挙ポスターの定石）。

## 5. 人物写真の扱い

- **removebg画像は必ずアルファ値bboxでクロップしてから使う**（`crop-people-images.js`）。透明余白込みでcontain-fitすると人物が意図せず縮む（実測：最悪48%しか写っていなかった）。
- 写真の構図（バスト/全身）とボックスの縦横比を合わせる。バスト写真を全身用の縦長ボックスに入れると顔が変な位置に浮く。
- **顔にテキストを重ねない**。視線・ジェスチャーの方向に余白／文字を置く。
- ポーズとテーマを一致させる：上向きの真剣な顔=選挙ポスター、かめはめ波=価格を手元に、丸を抱える=そこに商品画像。**写真が持っている物語を核にテーマを立てる**と強い。

## 6. pptxgenjs 技術Tips（ハマりどころ）

| 現象 | 原因と対策 |
|---|---|
| PowerPointが「ファイル破損」と言う | ①シェイプの `w`/`h` が**負数**（線の座標計算は `Math.min`/`Math.abs` で正規化必須）②**`shadow: { type: 'inner' }`** — pptxgenjsが壊れたXMLを吐く。**inner shadowは使用禁止** |
| どのスライドが破損源か分からない | **二分探索**：スライドブロックを1枚ずつ個別pptxに書き出し、PowerShell COMでループopenして特定する（実績あり・数分で特定できる） |
| テキストが勝手に折り返す・箱からはみ出す | spAutoFitが効かない。`wrap: false` + 十分な `w`、大フォント複数行は行数×サイズから箱を逆算 |
| 縦書き（`vert: 'eaVert'`）が蛇行する | 促音「っ」の処理が不安定。**1文字ずつ個別テキストボックス**で縦に並べる（促音は0.75倍サイズ、句読点「、。」はマス高0.5倍で`align:'right'/valign:'top'`＝縦組の右上寄せ、長音「ー〜」は`rotate:90`）。実装は`generate-nine.js`の`vText()`を再利用 |
| 集中線がショボい/引き出し線に見える | 中心から生やすのではなく**四辺の等間隔点→焦点への収束線**で引く。太さ・透明度を2〜3段階混ぜる |
| 星形バースト(star16等)が安っぽい | プリセット星形のフラット塗りはクリップアート感が出る。使わないか、写真素材で代替 |
| 三角形の大量重ねで放射線 | **禁止**（ユーザー指示）。線シェイプ十数本で表現 |
| 枠画像が中身を覆う | 枠の透明窓のbboxをpngjsで実測し、**窓がコンテンツ領域に一致する非等倍ストレッチ**を数式で計算 |
| 生成できない（EBUSY） | PowerPointがファイルを開いている。ユーザー編集中でないか確認してから閉じる |
| `sizing:{type:'cover'}`が写真を引き伸ばす | pptxgenjsのcover/containは**実画像サイズを読まずボックスサイズをそのまま「画像サイズ」として計算する**バグ（`imgWidth=cx`のFIXMEコメントあり）。ボックスと画像のアスペクト比が近い時は目立たないが、離れているほど露骨に伸びる。`getPngSize`で実寸を取得し、top-levelの`w`/`h`に実アスペクト比を渡してから`sizing.w/h`でボックスを指定する`addCover()`ヘルパー（`generate-nine.js`）を必ず使う |

### 枠画像フィットの計算式
```js
// fL,fR,fT,fB = 窓のbbox比率（実測）、border = 見せたい縁の太さ(in)
const boxW = (W - 2*border) / (fR - fL);
const boxH = (H - 2*border) / (fB - fT);
const boxX = border - fL * boxW;
const boxY = border - fT * boxH;
```

## 7. 運用ルール（このプロジェクト固有だが重要）

- **一度ユーザーに見せたpptxは手動編集されている前提**で扱う。再生成前に確認する。
  - **例外なし。** PowerPointがファイルを開いていて再生成できない（EBUSY / Device or resource busy）場合は、**理由を決めつけず必ずユーザーに確認してから閉じる**。「自分が直前にエクスポートした残骸だろう」という推測だけでStop-Processするな——ユーザーが今まさに手動編集中の可能性を常に疑え。過去に「同一セッション内の修正指示中なら確認なしで閉じてよい」という例外を設けていたが、これが原因でユーザーの手動編集（いいね・コメントアイコンの手直し）を無断で握りつぶす事故を起こした。二度とやるな。
  - 迷ったら「PowerPointが開いていて再生成できません。編集中でしたら保存してから教えてください」と聞くだけでいい。それだけで防げた事故だった。
  - **再生成・上書き前に、必ず `scripts/check-flyer-hash.ps1` で照合する**（`powershell -File scripts/check-flyer-hash.ps1 -File "flyers\xxx.pptx"`）。不一致（=手動編集の可能性）なら絶対に上書きせず、ユーザーに確認する
  - **コード強制（2026-07-07〜）**：全生成スクリプトは `pptx.writeFile()` を直接呼ばず、`pptx-safe-write.js` の `safeWriteFile(pptx, fileName)` を使う。承認済みハッシュ（`.approved-hashes.json`）と現物のSHA256が食い違う場合、**Claudeの判断に関係なくスクリプトが例外を投げて書き込みを拒否する**（ASKED→FORCEDの変換）。新しい生成スクリプトを書くときも必ずこれを使うこと。ただし「承認記録なし（NO_RECORD）」の判断はコードではできないので、そこは引き続き上記のハッシュ照合結果を見た上での口頭確認に頼る
  - ユーザーが承認したら `powershell -File scripts/approve-flyer.ps1 -Files "flyers\xxx.pptx"` でハッシュ記録 → `git add` + `git commit` でスナップショット化
- 手動編集を新スクリプトに取り込むときは、**元スクリプトではなく編集後の状態**（XML解析 or 確定JPG）から座標・サイズを再現する。
- 確定版のJPG（`flyers/*.jpg`）は正解のスナップショット。迷ったら見比べる。
- スライド番号と生成ブロックの対応をスクリプト冒頭コメントに必ず書く。

## 7.6 実戦で効いた「勝ちパターン」（ユーザー高評価の実績）

- **特大一文字**（和モダンの「旨」320pt・透かし気味）＝ 実測で最強クラスのインパクト。1文字に意味を凝縮して画面の核にする。朱印（本気/必勝）と組むとさらに締まる
- **持ちポーズ×商品オーバーレイ**：手で輪を作る/構える写真に商品画像を重ねて「持たせる」と一気に物語が生まれる（号外の手の中のホットドッグ）。手の位置は実測で合わせ、商品は小さめ＋軽い回転
- **チーズの穴に価格**・**公約にメニュー**のような「モチーフへの情報埋め込み」は毎回高評価。装飾を足すよりも情報の置き場所を発明する
- **本人の名前を入れる**（記事の実名・落款「保成」）と身内ウケが跳ねる。文化祭ポスターは「知ってる人が出てる」が最強のフック
- **写真は非破壊が原則**：円形cover-cropは頭や商品が切れて不評。円形の台紙（ellipse）の上に透過PNGをそのまま重ねる方が「ステッカー感」も出て強い

## 7.7 素材参照とスクリプト編集の罠

- **`peopleImgs[4]`のようなインデックス参照は禁止**。素材フォルダのファイル増減で全インデックスが黙ってずれ、複数スライドの写真が一斉に入れ替わる事故が起きた（実際に起きた）。`personByNum('1000019040')`のような**ファイル名解決**を使う
- **テキストの装飾記号（✦♥☺等）はフォント非対応だと黒い棒に化ける**。スパークルは`star4`シェイプ、ハートは図形で。★♪は和文フォントにあるので安全
- node -e をbashのダブルクォートで包んで置換スクリプトを書くとき、**テンプレートリテラルのバッククォートがシェルのコマンド置換として実行されて中身が消える**。複雑な置換はtmpファイル＋Editツールで行う。置換後は必ずgrepで着地確認（silent failする）
- 素材一覧は**cropped済みフォルダを直接scanする**（元フォルダをscanしてファイル名一致でcroppedに差し替える方式だと、croppedにしか無い画像がリストから漏れてpersonByNumが黙って見つからず、写真枠が空白になる。実際に発生）

## 8. テーマ発想のフレームワーク

1. **既存テーマと軸をずらす**：色（暖/寒）、時代（レトロ/未来）、文脈（食品広告/選挙/映画/コミック）
2. **持っている写真から逆算**（§5）
3. **「情報の言い換え」が成立するテーマは強い**（公約・指名手配・号外・処方箋…）
4. リサーチしてから作る：そのジャンルの**定石（記号）を3つ以上**入れると「それっぽさ」が出る
   （選挙なら：ひらがな名・たすき・公約・党名バッジ。ダイナーなら：チェッカー・ネオンサイン・星条旗色）
