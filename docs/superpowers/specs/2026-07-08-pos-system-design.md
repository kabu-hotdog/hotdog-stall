# レジシステム 設計仕様書

- 日付: 2026-07-08
- 対象: 旭川高専文化祭「おいしいホットドック屋さん」当日運用システム
- ステータス: 設計承認済み・実装計画待ち

## 背景・目的

文化祭当日、PC4台を使って「注文受付〜会計〜厨房連携〜受け渡し案内」をリアルタイムに連携させるレジシステムを作る。現金のみのシンプルな屋台だが、レジ・客表示・厨房・大画面案内（smartboard）の4画面が同じ注文状態を共有し、マクドナルドの受け取り案内のような体験を実現する。

配置場所: hotdog-stall リポジトリ内（新設 `pos/` サブフォルダ想定）。既存のフライヤー生成機能（`generate-*.js` 等）とは完全に独立したサブシステムとして共存する。

## スコープ

含む:
- POS画面（注文入力・会計・売上管理・受渡確認）
- 客表示画面（会計内容のお客様向け表示）
- 厨房画面（注文キュー・調理完了操作）
- smartboard画面（受け取り番号の大画面表示）
- 4画面をリアルタイム同期する中央サーバー

含まない（将来検討）:
- 電子決済（QRコード等）
- レシート印刷
- スタッフ別ログイン・権限管理
- 複数日/複数イベントにまたがるデータ管理（当日単発運用が前提）

## 前提条件

- 支払いは現金のみ
- 当日インターネット接続なしでも動作すること（LAN内で完結）
- LAN構築手段（ルーター購入かホットスポット利用か）は未確定 → ソフトウェア側は「サーバーPCのIPアドレスにブラウザで接続する」方式にしておき、どのLAN構成でも対応可能にする
- メニュー・価格は `.claude/skills/stall-info.md` を正とする（ホットドッグ200円／チーズホットドッグ300円、トッピング3種×2段階すべて無料）

## アーキテクチャ

**中央サーバー方式**: Node.js（Express + Socket.io）のサーバープロセスを4台のうち1台（レジ担当PC想定）で起動する。他の3台はブラウザで `http://<サーバーIP>:<port>/pos` `/customer` `/kitchen` `/smartboard` を開くだけで動作する。

- 注文データはサーバーのメモリ上で一元管理（単一の信頼できる状態源）
- 状態変化（新規注文・会計確定・調理完了・受渡完了・取消）はSocket.ioで全クライアントへ即時push
- 10〜15秒ごと、および重要イベント発生時にJSONファイルへスナップショットを書き出す。サーバー再起動時はこのJSONから状態を復元する
- クライアント（ブラウザ）はステートレス。リロード・クラッシュ時はサーバーから最新状態を再取得するだけで復旧する
- Socket.ioの自動再接続を利用し、切断中はUIに「オフライン」表示を出す

他アーキテクチャを検討したが不採用:
- 共有フォルダ＋ファイルポーリング方式: リアルタイム性が低く、同時書き込みで壊れるリスクがあるため不採用
- クラウドサービス（Firebase等）: 「ネットなしでも動作」の前提と矛盾するため不採用

## データモデル

```
Order {
  id: number,          // 注文番号。採番規則は下記「注文番号ルール」参照
  items: [
    {
      product: "hotdog" | "cheese_hotdog",
      toppings: {
        ketchup: "none" | "normal" | "extra",
        mustard: "none" | "normal" | "extra",
        mayo:    "none" | "normal" | "extra",
      }
    }, ...
  ],
  total: number,        // 合計金額
  received: number,     // 預かり金額
  change: number,       // おつり
  status: "cart" | "paid" | "cooking" | "ready" | "handed" | "cancelled",
  createdAt, paidAt, readyAt, handedAt
}
```

- ホットドッグ1本ごとに `items` 配列の要素として個別にトッピングを指定できる（本数分の粒度）
- チーズホットドッグはチーズ50g固定（量選択なし、トッピング3種の可否のみ選択）

### 注文番号ルール

- 新規注文（会計確定＝`paid`遷移時）に番号を発行する
- 採番時点で未受渡（`paid` / `cooking` / `ready` のいずれか）の注文が0件なら1から、それ以外は直近発行番号+1
- これにより行列が捌けるたびに番号が1にリセットされ、番号が大きくなりすぎない

## 状態遷移

```
cart（POS内で編集中、番号未発行）
  → paid（会計確定・番号発行・厨房画面に表示）
  → cooking（任意ステップ。厨房が着手を明示したい場合のみ使用）
  → ready（厨房が「調理完了」→ smartboardに表示）
  → handed（POSで「受渡済」→ smartboardから消える）

cart → cancelled（会計前に破棄。自由に編集・削除可能）
paid / cooking / ready → cancelled（POSの「取消」ボタンでキャンセル。厨房・smartboardからも即座に消える）
```

厨房操作を1ステップ（未着手→完了のみ）にするか、2ステップ（`cooking`を明示的に使う）にするかは実装時に決定してよい。初期実装は1ステップ（`paid`→`ready`）とし、シンプルさを優先する。

## 画面仕様

### POS（レジ）
- 商品追加: ホットドッグ／チーズホットドッグを選び、本数分のトッピングを個別選択してカートに追加
- 会計前はカート内容を自由に編集・削除可能
- 合計金額表示 → 預り金入力 → おつり自動計算 → 会計確定で注文番号発行、厨房へ送信
- 会計確定後も「取消」ボタンでキャンセル可能（厨房・smartboardにも即反映）
- 「受渡済」操作: `ready` 状態の注文一覧から選んで「受渡済」ボタン → `handed` に遷移、smartboardから消える
- 売上ボタン: 別ビューで累計売上額・販売本数（リアルタイム）、時間帯別・注文履歴一覧を表示
- スタッフ識別・ログインは不要（共用端末として誰でも使える）

### 客表示
- レジ裏に設置。会計中の内容（商品内訳・合計金額・預り金・おつり）をお客様向けに大きく表示するだけの読み取り専用画面

### 厨房
- `paid`（および`cooking`）状態の注文をキュー表示。番号・本数・トッピング内訳を表示
- 「調理完了」ボタンで `ready` に遷移

### smartboard
- `ready` 状態の注文番号一覧を大画面に表示（マクドナルド風の受け取り案内）
- 音声アナウンスは行わない（厨房スタッフが肉声で呼び込みを行う運用のため、画面はあくまで補助表示）
- POSで「受渡済」が押されると該当番号が消える

## 障害対応・永続化

- サーバープロセスがクラッシュ/再起動した場合: 直近のJSONスナップショットから状態を復元
- クライアント（ブラウザ）がリロード/クラッシュした場合: 再読み込みでサーバーから最新状態を取得するのみで復旧、影響なし
- ネットワーク瞬断: Socket.ioの自動再接続を利用。切断中はUIに「オフライン」表示

## QA方針

- 本番前に4画面を実機（または同等構成）で通しリハーサルを最低1回実施（注文→会計→厨房→smartboard→受渡の一連の流れ）
- 混雑時想定: 複数注文が同時に `paid`〜`ready` に並ぶケースを確認
- 取消・修正操作が各画面に正しく反映されるか確認
- サーバー再起動時のJSON復元が正しく動作するか確認
